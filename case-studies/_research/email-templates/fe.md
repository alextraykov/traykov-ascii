# Email Templates — Frontend Engineering Notes

_Prototype: `workflow` branch. All line refs are current as of 2026-04-22._

---

## 1. Editor Architecture

The editor is **not** ProseMirror, Lexical, or Slate. It is a hand-rolled block model rendered through React + a `contentEditable` div per block. No third-party rich-text library is involved.

**State shape** (`types.ts:113`):

```
Document = Block[]
Block = TextBlock | ListBlock
TextBlock  → { id, type: 'paragraph'|'heading1'|…|'quote'|'code', inlines: Inline[] }
ListBlock  → { id, type: 'bulletedList'|'numberedList', items: ListItem[] }
Inline     → TextInline | VariableInline
```

The canonical `Document` type alias is `Block[]` (`types.ts:113`). Each block has a stable `id` generated by `generateBlockId()` (`types.ts:11`), which uses `Date.now()` plus a random suffix — fine for a prototype, fragile in production under rapid insertion.

`TemplateEditor` holds `document` as a `React.useState<Document>` (`TemplateEditor.tsx:646`). Block mutations go through `handleBlockChange`, `addBlockAfter`, `removeBlock`, and `convertBlock`. The parent page (`TemplateEditorPage`) receives the full document on every change via an `onChange` prop.

DOM management is **fully imperative inside `InlineRenderer`**. A `useLayoutEffect` at `TemplateEditor.tsx:95` rebuilds the `contentEditable` div's innerHTML whenever `inlines` changes, specifically to avoid the classic React reconciliation / `contentEditable` conflict. An `isTypingRef` flag (`TemplateEditor.tsx:75`) suppresses the layout effect while the user is actively typing so keystrokes are not clobbered by React re-renders. The two code paths are: pure-text blocks (sets `innerText` directly, `TemplateEditor.tsx:101`) and blocks containing variable chips (full `innerHTML` rebuild, `TemplateEditor.tsx:124`).

Block reordering uses Framer Motion's `Reorder.Group` / `Reorder.Item` (`TemplateEditor.tsx:1312–1350`). Drag-to-reorder directly calls `setDocument` via `onReorder`.

## 2. Slash Commands

**Trigger detection** is in `InlineRenderer.handleInput` (`TemplateEditor.tsx:168`). On every `onInput` event the handler reads the text before the cursor from the DOM selection, then tests it against `(?:^|\s)\/(\S*)$`. If matched, it records the caret rect and calls `onOpenSlashMenu` with the viewport position and the query string (the text typed after `/`).

The `SlashCommandMenu` renders into `document.body` via `createPortal` (`SlashCommandMenu.tsx:253`), positioned by `style={{ top, left }}` from the caret `getBoundingClientRect`. It filters the 9 static `slashCommands` definitions by `label` and `description` (`SlashCommandMenu.tsx:130–138`).

Keyboard handling attaches a capturing `document.addEventListener('keydown', …, true)` (`SlashCommandMenu.tsx:176`) so it intercepts before the editor. Arrow keys move `selectedIndex`; Enter calls `onSelect`; Escape calls `onClose`; Tab completes to the first match's remainder (`SlashCommandMenu.tsx:216`). The autocomplete ghost hint (typed query + dimmed remainder + "Tab" key badge) is computed in `autocompleteSuggestion` (`SlashCommandMenu.tsx:141`).

On selection, `handleSlashCommand` (`TemplateEditor.tsx:1059`) strips the `/ + query` chars from the block's inlines before converting the block type. Conversion dispatches `convertBlock`. The `/variable` command (`action: 'insert-variable'`) skips conversion and instead opens the variable picker at the same position.

There is no `image` or `divider` slash command in the current prototype — only text, headings, lists, quote, code, and variable.

**Markdown shortcuts** are handled separately in `BlockRenderer.handleInlinesChangeWithMarkdown` (`TemplateEditor.tsx:436`). Typing `# `, `## `, `### `, `- `, `1. `, `> `, or ` ``` ` at the start of a paragraph fires a `CustomEvent('markdown-convert')` on `window` (`TemplateEditor.tsx:459`), which the editor root catches at `TemplateEditor.tsx:726` and calls `convertBlock`.

## 3. Variable Tokens

**Trigger**: typing `{{` in any block fires `onOpenVariablePicker` via the same `handleInput` caret scan (`TemplateEditor.tsx:207`).

The `VariablePicker` also portals to `document.body` (`VariablePicker.tsx:183`) with a search `<input>` that autofocuses on open (`VariablePicker.tsx:51`). Tables and fields come from `mockQuickbase.ts` (4 tables, 24 fields total). On selection, `handleVariableSelect` (`TemplateEditor.tsx:1142`) scans the DOM's `innerText` for the last `{{`, splits it out, and rebuilds the inlines array as `[...before, VariableInline, { type:'text', text:'' }]`.

**DOM rendering of chips**: inside `useLayoutEffect`, each `VariableInline` becomes a `<span contenteditable="false" data-variable="true" role="button" tabindex="0">` with a child label span and a `×` remove button (`TemplateEditor.tsx:129–150`). The chip is atomic — the `contenteditable=false` attribute prevents the cursor from entering it.

**Backspace deletion** of a chip is handled in `InlineRenderer.handleKeyDown` (`TemplateEditor.tsx:232`): if the cursor is at offset 0 of a text node whose `previousSibling` has `data-variable="true"`, the event is prevented and the matching variable is spliced from the inlines array. Chip removal via the `×` button uses event delegation on the `InlineRenderer` click handler (`TemplateEditor.tsx:302`).

**Serialization**: `VariableInline` serializes to `{{Label}}` via `serializeInlines` (`types.ts:123`). The label is the Quickbase field label string — not the field ID — so round-tripping through `deserializeFromTemplateString` produces `tableId: 'unknown', fieldId: 'unknown'` stubs (`types.ts:193`). The IDs are only preserved in the in-memory `Document` object.

## 4. Serialization

Format is **Markdown with `{{variable}}` extensions**. There is no JSON or HTML format; the sole interchange representation is a Markdown-dialect string.

`serializeToTemplateString(document: Document): string` (`types.ts:166`) joins `serializeBlock` results with `\n\n`. `serializeBlock` (`types.ts:137`) outputs:
- Headings as `#`/`##`/`###`
- Quote as `>`
- Code as triple-backtick fences
- Lists as `- ` or `1. ` prefixed lines
- Paragraph as bare text

`deserializeFromTemplateString` (`types.ts:215`) is a line-by-line parser: it checks prefix patterns in order (heading3 before heading2 before heading1 to avoid prefix collision), handles code fence multi-line capture, and groups consecutive list lines. Empty lines are skipped; anything else becomes a paragraph. It calls `parseInlines` to split `{{...}}` markers out of each text string.

The draft-detection in `TemplateEditorPage` serializes on every `onChange` and diffs the serialized string against `initialDocumentRef` (`TemplateEditorPage.tsx:67–76`). The draft badge appears when the two strings diverge.

## 5. Version History

There are **two separate version components** in the codebase with different scopes:

`VersionHistoryPopover` (`VersionHistoryPopover/VersionHistoryPopover.tsx`) is a Radix `Popover` with a `Command` (cmdk) list inside. It shows numbered versions (`Version 1`, `Version 2`, …) with relative timestamps, a "Published" globe badge for the deployed version, and hover-revealed Revert (↺) and "Preview in new tab" (↗) icon buttons. Clicking a row calls `onPreview` and closes the popover. There is no diff view — each version is a full opaque snapshot. The `TemplateVersion` type (`VersionHistoryPopover.tsx:29`) carries only `id`, `version`, `timestamp`, and `isDeployed`. No content payload is stored in the prototype.

`VersionPanel` (`VersionPanel/VersionPanel.tsx`) is a richer, slide-in panel used by the Builder page. It has a timeline-dot UI, source attribution icons (Bot for AI-generated, Pencil for user edits, Rocket for publish events), Preview / Revert hover actions, and a "Return to current" affordance when a historical version is being previewed. Its data is fully mocked in the component (`VersionPanel.tsx:41–93`).

Neither component implements an actual diff. Revert semantics (per the flow diagram) are "restore version as a new draft" — no branching model.

## 6. Toolbar

The page toolbar (`TemplateEditorHeader`) sits at the top of the page and contains:
- Left: breadcrumb (Email Notifications / template name with inline edit via `isEditingName` state, `TemplateEditorHeader.tsx:47`)
- Draft badge: `<span class="template-header__draft-tag">Draft</span>` rendered conditionally on `isDraft` (`TemplateEditorHeader.tsx:97`)
- Right icon group: Duplicate (Copy icon) and Delete (Trash2 icon), both tooltip-wrapped
- "Send test email" secondary button
- "Publish" primary button, disabled when `isPublishDisabled` (requires `publishConfig.name` and `publishConfig.tableId`, `TemplateEditorPage.tsx:65`)

There is **no inline formatting toolbar** (no Bold/Italic/Underline buttons). The editor exposes no `execCommand` or selection-based formatting. Block-type changes happen exclusively through slash commands or markdown shortcuts — there is no floating toolbar on text selection.

`TemplateToolbar` (`TemplateToolbar/TemplateToolbar.tsx`) is a separate, lighter toolbar component with breadcrumbs + "Describe with AI" + Publish, but is not the one used in `TemplateEditorPage` — the page uses `TemplateEditorHeader` directly.

No `Cmd+B` / `Cmd+I` shortcuts are implemented.

## 7. Performance

The prototype makes **no attempt at virtualization**. `document.map(...)` renders every block unconditionally (`TemplateEditor.tsx:1322`). Framer Motion `Reorder.Group` wraps the list, which adds per-item layout animations.

At 10k blocks this would be unusable. The contentEditable-per-block approach means 10k `[contenteditable]` DOM nodes are live simultaneously. There is no windowing, no deferred rendering, no intersection observer, and no block count guard.

The `isTypingRef` pattern does prevent redundant DOM syncs during rapid typing, but this is a micro-optimization irrelevant to the structural problem.

## 8. Email-Specific Fields

`EmailHeaderFields` (`EmailHeaderFields/EmailHeaderFields.tsx`) renders three `<input>` fields:
- **From** — `type="text"`, `id="email-from"`, label-associated, plain text (sender display name + address in a single field)
- **Reply-To** — `type="email"`, `id="email-reply-to"`, browser-validated email
- **Subject** — `type="text"`, `id="email-subject"`, with an adjacent AI sparkle button (`onGenerateSubject`) that is stubbed with a `console.log` in the page

The subject field has a Sparkles AI-generate affordance (`EmailHeaderFields.tsx:97`).

There is no **preview HTML** render — no live email preview pane, no HTML output, no mobile vs. desktop preview toggle. The `showPreview` prop on `TemplateEditor` renders only the raw serialized Markdown string, not rendered HTML (`TemplateEditor.tsx:1375`).

The **send test email** affordance is in the header ("Send test email" button, `TemplateEditorHeader.tsx:141`) and also as a `Button` in the config panel footer (`TemplateEditorPage.tsx:253`). The handler validates that `from` and `subject` are non-empty before proceeding, showing a `toast.warning` if not (`TemplateEditorPage.tsx:82–91`). The actual send is stubbed with `console.log`.

The right panel (`NotificationConfigPanel`) handles notification delivery settings (table, trigger conditions, recipients) — not rendered content.

## 9. Accessibility

**Positive signals:**
- Each `InlineRenderer` div sets `role="textbox" aria-multiline="false"` (`TemplateEditor.tsx:349`)
- The `Reorder.Group` has `role="document" aria-label="Template editor"` (`TemplateEditor.tsx:1319`)
- Variable chips have `role="button" tabindex="0"` and `aria-label="Remove {label}"` on the remove button (`TemplateEditor.tsx:135–146`)
- `SlashCommandMenu` has `role="listbox" aria-label="Slash commands"` and each item has `role="option" aria-selected` (`SlashCommandMenu.tsx:263–299`)
- `VariablePicker` has `role="listbox" aria-label="Variable picker"` (`VariablePicker.tsx:194`)
- All toolbar icon buttons are tooltip-wrapped and have `aria-label` attributes
- Form labels are properly associated via `htmlFor`/`id` pairs in `EmailHeaderFields`

**Gaps:**
- The block list has no `aria-activedescendant` pointing to the focused block
- `aria-multiline="false"` is incorrect for a rich-text block editor — it should be `true` on multi-line blocks
- The `Reorder.Item` drag handles (`GripVertical`) have `aria-label="Drag to reorder"` but are `<div>` elements, not `<button>`, so they are not keyboard-accessible for drag initiation
- No `aria-live` region announces block insertions, deletions, or type conversions to screen readers
- `contentEditable` blocks don't have `aria-label` identifying their block type (e.g. "Heading 1 block")
- Variable chips inside `contentEditable` are invisible to the accessibility tree because screen readers cannot traverse into `contenteditable=false` children reliably

## 10. Production-Port Concerns

**Saving cadence**: The prototype has no autosave. `isDraft` is a local boolean toggled by string diff. There is no debounced save, no periodic flush, no "saving…" indicator, and no localStorage fallback. The initial document is captured on the first `onChange` call (`TemplateEditorPage.tsx:71`), but subsequent saves are entirely user-initiated via Publish.

**Conflict resolution**: Not modeled. No optimistic locking, no last-write-wins indicator, no collaborative awareness. The `TemplateVersion` type has no author field.

**Template engine parity**: The serialization format is Markdown + `{{Label}}` tokens. A production email template engine (Handlebars, Liquid, Jinja2, or QB's own engine) would need to consume `{{field_id}}` or `{{tableId.fieldId}}` syntax, not human-readable labels. The current `VariableInline` stores both `tableId`/`fieldId` and `label`, but `serializeInlines` only emits the label (`types.ts:129`). The in-memory Document has the IDs; the serialized string loses them. The deserializer restores only stub IDs. This means the client's preview string (`{{Full Name}}`) and what the server renderer would need (`{{field-1}}` or a QB formula) are mismatched. Production must either serialize IDs or maintain a label-to-ID resolution pass at send time.

**Block ID stability**: `generateBlockId()` uses `Date.now()` plus random chars. In a real persistence layer, block IDs need to be server-assigned UUIDs for stable diffs and version history anchoring.

**Version storage**: Both `VersionHistoryPopover` and `VersionPanel` are fully mocked. Neither is wired to a real API or local version stack. Revert calls only fire `onRevert` callbacks; no document state is actually restored in the prototype.

**No undo/redo stack**: There is no `useUndoStack` or `useHistory` hook. Ctrl+Z falls through to the browser's native `contentEditable` undo for the active block only — it cannot undo block insertions, deletions, or type conversions.
