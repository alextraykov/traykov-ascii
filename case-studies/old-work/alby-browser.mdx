## Summary

Alby Browser was listed in my old portfolio as work connected to Alby's browser wallet. The product category is clear: a browser extension that lets people use Bitcoin Lightning directly on websites, with payments and identity flows living closer to the page where the action happens.

## Problem

Lightning payments on the web can become fragmented fast. A user might see a payment request in one surface, approve it in another, and then return to the browser after the context has already gone stale.

The browser extension model reduces that distance. The product work needed to make wallet actions feel immediate while still keeping enough friction around trust, site permissions, and spending allowances.

## Approach

The important design tension was convenience versus control. A browser wallet should make payments feel native to the web, but it should not hide the fact that value is moving.

That points to a few interface priorities:

- A clear connection state between site and wallet.
- Explicit spending limits for trusted sites.
- Fast approvals for familiar flows.
- Stronger review moments when the destination or allowance changes.

```mermaid
flowchart LR
  Site[Website asks for payment] --> Extension[Browser wallet surface]
  Extension --> Review[Review amount and site]
  Review --> Approve[Approve or reject]
  Approve --> Return[Return to web context]
```

## Outcome

The case study translates the old portfolio item into a product-design story about reducing payment context switching while preserving user control.

## What I would sharpen now

- Make site permissions easier to audit after the first approval.
- Separate one-time payments from standing budgets visually.
- Use plain-language payment states for non-technical users.
