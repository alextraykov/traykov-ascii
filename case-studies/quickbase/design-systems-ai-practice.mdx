## I briefly thought documentation was the answer

I authored an index for 23 production-spec'd Bebop components and, for a moment, this felt like the sensible answer to AI-generated UI: give the machine better documentation and it will stop making things up.

Better documentation helped. It did not perform miracles.

An agent can read component anatomy, token names, and usage rules and still produce something plausible but wrong. The work across Bebop 2, Pave, and our internal AI experiments became an attempt to make the system readable *and* put human review where the system could drift. “AI owns toil. Humans hold judgment” was the operating line. The second half did most of the work.

## Whose work this was

Several people contributed to this work:

- Bebop 2 was cross-functional.
- The Bebop 2 Component Index was my authored contribution.
- Foundations work was broader team work, with Chavdar owning key foundations material.
- Blinq component analysis was authored by Nick Vanselow; my contribution was using it to de-risk rebrand and dark-theme/token decisions.
- Mini-Hack was co-driven with Stasha.

My own work was the component index, applying the material to Pave, and the AI workflow experiments below. The attribution is fussy because design-system case studies have a bad habit of turning shared infrastructure into one person's heroic origin story.

## Twenty-three components, written down properly

I authored the Bebop 2 Component Index: 23 production-spec'd components plus token and usage context.

The index explained anatomy, variants, props, behaviour, and linked specs. Designers and tools could follow the component contract instead of squinting at a screenshot and improvising the rest.

The count in the source was:

- Interactive: 3.
- Form controls: 8.
- Overlays: 4.
- Navigation: 3.
- Indicators and feedback: 3.
- Layout and structure: 2.
- Total: 23.

The count only mattered because names connected to React exports and Figma names, entries deep-linked to spec frames and code references, and maintenance had an owner with a quarterly expectation. Otherwise “23 documented components” would become “23 increasingly historical opinions.”

## Rules a machine could fail against

The foundations turned visual decisions into constraints that software and agents could follow.

The source foundations include:

- 2px spacing grid.
- Radius scale.
- 9 typography tokens.
- 9 palettes with 8 shades each.
- Semantic light/dark tokens.
- 5 elevation levels.
- 3 motion durations/easings.

The non-negotiable rules read like a lint layer for design:

- Never hardcode colors.
- Use spacing only from the grid.
- Define all interactive states.
- Keep touch targets at 44x44.
- Use fixed elevation and motion rules.

Once written this way, a wrong value could fail a rule instead of waiting for someone with a good eye to notice it in review. The review still mattered. It just no longer had to catch every invented shade of grey by hand.

## The rebrand was the unpleasantly useful test

Pave's prototype environment forced work through semantic tokens instead of one-off values. Then Blinq became Pave, which was the kind of disruption that tells you whether the system is real or just nicely documented.

Semantic colours, the Pave accent, oatmeal backgrounds, logos, and fonts changed without restyling individual components. Literal brand use did not get diplomatic immunity from contrast: raw Pave Yellow measured 1.11:1 on white, so UI text used an amber at 5.13:1; red moved from 3.5:1 to 5.47:1.

Nick’s component analysis also showed us which changes would travel furthest:

- Button appeared in 76 files.
- Modal appeared in 30 files.
- SearchInput appeared in 13 files.
- Tooltip appeared in 8 files.
- ConfirmDialog appeared in 7 files.
- ColorPalette appeared in 7 files.
- SegmentedControl appeared in 5 files.
- FlyoutMenu appeared in 4 files.
- Lucide appeared 257 times across 80 unique icons.

Nick authored the analysis. I used it to decide where token and rebrand changes would travel furthest, starting with the button used in 76 files rather than whichever component happened to look most fun that afternoon.

## Helping designers get useful answers from AI

One early enablement document helped designers fit useful Quickbase context inside ChatGPT's 1,500-character custom-instruction limit.

The template asked for Confluence or Jira structure, risks, tradeoffs, open questions, and clarifying questions. Very glamorous. It was still more useful than pretending an empty chat box was a strategy.

## Keeping human judgment in the loop

The Design Spine was an internal Claude-skills process plugin with seven phases:

- Intake.
- Enrichment.
- Diverge.
- Converge.
- Resolve.
- Evaluate.
- Document and close.

AI handled repetitive translation between the brief, exploration, decision, and documentation. A designer approved the points where the work changed direction or became a decision.

Design Spine reached v0.4.1 and included a drift gate. It remained an internal experiment. The version number is precise; the organisational outcome is not. I do not yet have enough usage data to say how well it worked beyond the people close to it.

## Making room for experiments

The Innovation Register was simple by design. Its submission template asked for the pattern, user problem, what was new, whether the decision could be reversed, rollback path, blockers, expected measure, effort, confidence, owner, and decision state.

We designed the Mini-Hack programme around these targets:

- 2-week prep.
- 3-day hack.
- 1-week adoption review.
- 4-week incubation.
- Goals: 60% of teams producing working prototypes, 2 prototypes adopted/incubated per quarter, 1 production initiative per quarter.

These were targets, not measured results.

## Trying a Figma-to-code workflow

I also proposed an early Bebop 2 to code workflow through Figma MCP.

A naive design-to-code path sees pixels and misses anatomy, variants, behaviour specs, and composition rules. My proposal added those layers and a designer review gate before code output could be treated as useful.

It remained a proof of concept and did not ship as a platform capability.

## The part I have not earned yet

I can show the component index, the rules, the rebrand, the Design Spine prototype, the Innovation Register, and the Figma-to-code proposal. I cannot yet show that design-to-code attempts pass review more often or that teams ship faster because of them.

That is the next test: usage of the index, review pass rate, review time, and which teams can use the system without its authors hovering nearby. Until then, this is a promising practice with good artifacts - not a transformation story.
